# 🎯 목적

이 스킬의 목적은 기능 추가 및 수정 작업에서 보안 검토가 뒤로 밀리지 않도록,
**개발 / 구축 / 운영 전반의 보안 점검 항목을 기본 절차로 강제**하는 것이다.

UI든 API든, DB든 외부 연동이든 간에
**보안은 부가 옵션이 아니라 기본 요구사항**으로 적용한다.

---

# ❗ 기본 원칙

1. 보안은 구현 후 확인이 아니라 **설계 단계부터 반영**한다.
2. 입력은 항상 신뢰하지 않는다.
3. 인증과 인가는 별개로 검토한다.
4. 민감 정보는 저장 / 전송 / 로그 / 응답에서 모두 분리해서 본다.
5. 실패 시에도 안전한 방향으로 동작해야 한다. (`fail closed` 우선)

---

# ✅ 기능 추가 및 수정 시 필수 확인 순서

기능 작업 전 반드시 아래 순서를 따른다.

1. 외부 입력 지점 확인
2. 인증 필요 여부 확인
3. 인가 필요 여부 확인
4. 데이터 노출 가능성 확인
5. 외부 연동 실패 처리 확인
6. 로그/모니터링 민감정보 확인
7. 테스트 가능 여부 확인

👉 기능 구현은 이 검토 후에 진행한다.

---

# 🔍 필수 보안 점검 항목

## 1) 사용자 입력 검증

반드시 확인:

- query / body / params / headers / cookies / searchParams / formData 검증 여부
- 타입만 맞는지 말고 **허용 범위 / 길이 / 형식 / enum / null 처리**까지 확인
- 서버에서 다시 검증하는지 확인
- 파일 업로드가 있으면 MIME / 크기 / 확장자 / 개수 제한 확인

질문:

- 이 입력은 어디서 들어오는가?
- 클라이언트 검증만 있고 서버 검증이 빠진 것은 아닌가?
- 예상하지 않은 값이 들어왔을 때 안전하게 실패하는가?

---

## 2) 인증 / 인가

반드시 구분해서 본다.

- 인증: 이 사용자가 누구인지 확인했는가?
- 인가: 이 사용자가 **이 작업을 해도 되는지** 확인했는가?

반드시 확인:

- 로그인 여부 확인이 필요한 기능인가?
- 본인 소유 리소스만 접근/수정/삭제 가능한가?
- 관리자/운영자 권한이 필요한 기능인가?
- 프론트에서 숨기는 것만으로 권한을 처리하고 있지 않은가?

질문:

- URL만 알면 남의 데이터가 보이지 않는가?
- UI에서 버튼이 안 보여도 API 직접 호출로 우회되지 않는가?

---

## 3) 민감 정보 노출

민감 정보 예시:

- 토큰
- 세션 ID
- API Key
- 이메일 / 전화번호 / 위치 정보
- 내부 에러 상세
- 운영용 식별자

반드시 확인:

- 응답 JSON에 불필요한 내부 필드가 포함되지 않는가?
- 클라이언트 번들에 서버 비밀이 들어가지 않는가?
- 에러 메시지에 내부 구현/스택/환경 정보가 노출되지 않는가?
- 로그에 개인정보/토큰이 찍히지 않는가?

---

## 4) 대표적인 취약점 점검

### SQL Injection
- 문자열 결합 쿼리 사용 여부
- ORM이라도 raw query 사용 시 파라미터 바인딩 여부

### XSS
- 사용자 입력을 HTML로 직접 렌더링하는지
- dangerouslySetInnerHTML / raw HTML 주입 여부
- URL, 링크, title, markdown 렌더링에서 스크립트 삽입 가능성

### CSRF
- 상태 변경 요청이 쿠키 기반 인증이라면 CSRF 보호 여부
- Origin / SameSite / CSRF token 전략 확인

### SSRF / Open Redirect
- 서버가 외부 URL을 받아 요청하는 구조인지
- redirect next/url 파라미터가 상대 경로/허용 도메인으로 제한되는지

### 파일/미디어 처리
- 업로드 파일이 실행 가능 콘텐츠로 바뀌지 않는지
- 이미지/문서 파싱 중 위험 경로가 없는지

# 🌐 외부 API / 네트워크 호출

외부 연동은 기능보다 실패 전략을 먼저 본다.

반드시 확인:

- timeout 존재 여부
- retry 존재 여부 (무한 재시도 금지)
- fallback 전략 존재 여부
- rate limit / quota 초과 시 동작
- 에러 시 사용자에게 어떤 상태로 보이는지

질문:

- 외부 API 실패가 전체 기능 실패로 번지지 않는가?
- fallback이 있더라도 잘못된 데이터를 정상처럼 보여주지 않는가?

---

# 🧾 로그 / 모니터링 / 운영

반드시 확인:

- 로그에 개인정보, 토큰, 쿠키, Authorization header가 포함되지 않는가?
- 운영 로그가 너무 자세해서 내부 구조를 노출하지 않는가?
- 실패 원인 추적은 가능하지만 민감정보는 제거됐는가?

운영 관점 체크:

- 환경 변수 누락 시 안전하게 실패하는가?
- 보안 헤더가 기존 동작에서 사라지지 않았는가?
- 캐시가 민감 응답을 공유하지 않는가?

---

# 🧱 설계 원칙

## ❌ 잘못된 보안 처리

- 프론트에서만 막기
- 에러 나면 그냥 빈값 반환
- 민감 필드 전체 응답 후 UI에서만 숨기기
- 외부 API 실패를 조용히 삼키기
- 로컬에서는 되니까 넘어가기

---

## ✅ 올바른 보안 처리

- 서버에서 계약 기반 검증
- 인증과 인가 분리 확인
- 최소 응답 원칙
- 외부 호출은 timeout / retry / fallback 명시
- 로그는 진단 가능하지만 민감정보 제거
- 실패 시 안전한 상태로 종료

# 📊 작업 유형별 추가 체크

## API / 서버 로직 변경
- 입력 검증
- 인증/인가
- 응답 최소화
- 에러 코드 구조화
- rate limit 필요 여부

## UI / 폼 변경
- 서버 검증 누락 여부
- 링크/HTML/XSS
- 민감정보 화면 노출 여부

## DB / 저장 구조 변경
- 소유권 체크
- migration 안전성
- 민감정보 저장 필요성
- 삭제/보존 정책

## 외부 API 연동
- timeout / retry / fallback
- secret 처리
- 응답 검증
- 사용자 안내 문구

# ⚙️ 출력 요구사항

기능 추가/수정 응답에는 가능하면 아래를 포함한다.

1. 어떤 입력을 검증했는지
2. 인증 / 인가 판단이 어디에 있는지
3. 민감정보 노출 위험이 없는지
4. 외부 호출 실패 시 어떻게 동작하는지
5. 어떤 테스트/검증으로 확인했는지

---

# 🔥 핵심 요약

보안은 따로 붙이는 기능이 아니다.
기능 추가와 수정이 들어가는 순간,
입력 검증 / 권한 / 노출 / 외부 호출 실패 / 로그까지 같이 봐야 한다.

보안 검토 없이 구현된 기능은 완료된 기능이 아니다.
